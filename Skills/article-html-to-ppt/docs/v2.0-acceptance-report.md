# MeowClaw PPTSmith v2.0 Acceptance Report

**Acceptance date:** 2026-07-20 (Asia/Shanghai)

## Verdict by scope

- **Standard:** engineering-complete with component-scoped readiness on the verified acceptance environment. `python_pptx` 1.0.2 is qualified for the accepted fixture and simple bound process diagrams; PptxGenJS 4.0.1 remains portability evidence on the same source semantic contracts.
- **LibreOffice Premium acceptance:** **final** on the recorded PptxGenJS 4.0.1 + LibreOffice 26.2.4.2 environment.
- **Native Microsoft PowerPoint compatibility:** **not verified**. The Premium result is not a claim of PowerPoint/Keynote pixel parity or identical behavior on other Office versions.

The public source repository retains this report and the executable acceptance fixtures/tests. Generated decks, rendered pages, manifests, and evidence bundles are maintained outside the public source tree.

## Standard acceptance baseline

The shared nine-slide fixture covers cover, judgment, evidence, editable table, native chart, simple process, comparison, implementation plan, and closing. Fresh isolated Standard runs established that both real Builders consume byte-identical PPT IR, Style Contract, and requirements contracts and preserve the same selected component routes. The canonical Standard deck is the `python_pptx` output; the PptxGenJS deck is secondary portability evidence, not a second canonical final.

The historical Standard claims remain structural and contract-based; they are not retroactively presented as a LibreOffice visual acceptance.

As of 2.0.5, the accepted simple process route requires one arrowed connector object with native endpoint binding (`a:stCxn`, `a:endCxn`, and an arrow end). The acceptance fixture does not prove that the minimal Builder can design every relationship visual. Complex architectures, layered systems, matrices, flywheels, ecosystems, stakeholder maps, and commercial staircases require a qualified component implementation and are blocked otherwise. Structural validity, native-text ratio, and nonblank rendering are necessary but not sufficient proof of professional information design.

## Premium acceptance result

The P1 Premium acceptance used a new isolated work/output root and the existing PptxGenJS portability route. The first pass intentionally stopped at the score gate because no rubric was supplied. That pass generated the exact deck and real LibreOffice render set for visual review. Only after reviewing the nine-slide contact sheet was a rubric created and cryptographically bound to that case/build/deck/PPTX/render report/render set. The same artifacts then continued through score, delivery preparation, and strict packaging.

| Gate | Result |
|---|---|
| Builder | PptxGenJS 4.0.1 |
| Renderer | LibreOffice 26.2.4.2 |
| Physical/rendered slides | 9 / 9 |
| Structural inspection/readback | passed; PPTX SHA bound |
| QA | pass; 0 fatal, 0 error, 0 warning |

| Build fallbacks | none; explicitly disclosed as an empty list |

The model visual review used all nine LibreOffice-rendered slides in one contact sheet. Reviewer provenance records a unique `review_id`, timestamp, current configured model version, and method. Every dimension has non-empty, slide-specific evidence. The reviewer noted clean rendering with no visible clipping, overlap, blank slide, broken glyph, or malformed object, while also noting sparse whitespace and basic component polish on several slides; the score therefore remained 15/18 rather than being inflated.

## Evidence binding and fail-closed behavior

Premium rubric trust is not inferred from a filename or a handwritten score. The scorer and delivery gate require matching:

- `case_id`
- `build_id`
- `deck_id`
- exact PPTX SHA-256
- canonical render-report SHA-256
- rendered-slide-set digest
- trusted reviewer provenance and non-empty evidence
- successful structural inspection whose PPTX SHA matches the delivered deck

Negative safety checks passed:

1. rubric with missing bindings was rejected;
2. stale render-set digest was rejected;
3. rubric replayed against another deck was rejected;
4. inspection SHA mismatch could not grant `final` (status was capped below final).

Renderer-unavailable and renderer-available-without-rubric branches remain fail-closed and produce no Premium final package.

## Evidence retention and privacy

The retained private Premium evidence bundle contains the case, bound rubric, build/capability/delivery manifests, structural inspection, QA report, render report, all nine rendered slide images, contact sheet, benchmark score, final Delivery Manifest, verification report, negative-safety transcript, ZIP integrity result, final deck, preview PDF, and SHA-256 inventory.

Generated acceptance artifacts are intentionally excluded from the public source tree. The retained bundle contains no `/Users`, `/private/tmp`, or `/tmp` path disclosures.

## Limitations

1. LibreOffice Premium acceptance applies only to the recorded Builder/renderer/version route and fixture.
2. Microsoft PowerPoint native opening, editing, rendering, transitions, and pixel parity have not been tested.
3. The deck passed the Premium gate but the visual reviewer still identified room for stronger density use and higher-end component craft.
4. `officecli`, `html_svg`, advanced animation/transitions, narration, timing, video, cloud publishing, and v2.1 remain out of scope.
